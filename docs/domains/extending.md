# Writing your own domain

A domain is a directory under `domains/` containing the wiring +
content the orchestrator needs to study a specific field. Nothing in
`core/` knows about your domain — it plugs in.

## Minimum viable domain

```
domains/<your_name>/
├── config.json            # required
├── context.md             # required (the model)
├── seed_tensions.json     # required (bootstrap for first cycle)
└── tools/                 # optional
    └── __init__.py
```

You can run a working cycle with just these. The other files
(`tension_to_category.json`, `corpus/`, custom tools) are optional.

## Step 1 — config.json

Use `domains/physics/config.json` as a template. Required fields:

```json
{
  "$schema": "../../config.schema.json",
  "domain": "your_name",
  "version": "0.1.0-alpha",
  "title": "Your Lab Title",
  "description": "What this lab studies / produces.",
  "model": {
    "context_file": "context.md",
    "seed_tensions": "seed_tensions.json"
  },
  "context": {
    "data_dir": "your_name",
    "max_turns": 25,
    "timeout_seconds": 1200
  },
  "tools": [
    { "name": "filesystem", "type": "builtin" },
    { "name": "python_exec", "type": "builtin" },
    { "name": "shell_exec", "type": "builtin" }
  ],
  "movements": {
    "agent":            { "enabled": true },
    "validate_seed":    { "enabled": true },
    "structural_check": { "enabled": true },
    "build_field":      { "enabled": true }
  }
}
```

The schema (`config.schema.json`) validates your config on load. Run
`dndlab inspect --domain your_name` to verify.

## Step 2 — context.md

This is the **model** of your domain — the agent reads it at every
cycle start. Sections that pay off:

1. **Identity** — who the agent is, what it's NOT (chat assistant /
   writer / etc.)
2. **The model** — the structural lens you apply (your dipoles, your
   axioms, your invariants)
3. **Anti-patterns** — errors already paid, do not repeat. List them
   concretely with examples; the agent reads these before each cycle
4. **The modus** — how to operate (expand → observe → cut → resultant,
   adapted to your domain)
5. **The bicono** — how to compile the bicono section in the report
   for your domain. Include a worked example.
6. **Output format** — what a cycle's report looks like
7. **What NOT to do** — guard rails the agent should not cross

Keep it dense. The whole file gets injected into the prompt at every
cycle, so every line costs tokens. Prune ruthlessly.

The physics context.md and editorial context.md are the references —
study both before writing yours, they show how the same skeleton
adapts to numeric vs semantic content.

## Step 3 — seed_tensions.json

Your starting field. Format:

```json
{
  "_meta": {
    "domain": "your_name",
    "description": "Initial tensions for the first cycle.",
    "ported_from": "..."
  },
  "tensioni": [
    {
      "tipo": "contraddizione",
      "id": "TENSION_FOO",
      "claim": "What the tension is about, in one sentence.",
      "intensita": 0.8,
      "porta": "novità",
      "potenziale": "alto",
      "stato": "aperto",
      "nota": "Optional context."
    }
  ],
  "direzione_iniziale": "What the lab should look at first."
}
```

Tension types (`tipo`) recognized by `seed_integrator` for direction
priority:

- `contraddizione` — A says X, B says not-X (highest priority)
- `confine_inesplorato` — boundary between regimes
- `simmetria_sospetta` — pattern is too clean, probably tautology
- `tensione_aperta` — generic open question
- `scoperta` — auto-generated by structural_check (anti-pattern detected)

Aim for 6–12 starting tensions. Too few → the agent has nothing to
compare; too many → the agent gets distracted. The lab will add and
remove tensions across cycles.

## Step 4 — tools (optional)

If your domain needs specialized tools beyond the universal
filesystem + python + bash, write them:

```python
# domains/your_name/tools/your_tool.py

def build(domain: str) -> dict:
    """Called by the orchestrator. Returns a ToolEntry."""

    def your_tool(arg: str) -> str:
        """The actual implementation."""
        return f"result of {arg}"

    return {
        "fn": your_tool,
        "schema": {
            "type": "function",
            "function": {
                "name": "your_tool",
                "description": "What this tool does. The LLM reads this.",
                "parameters": {
                    "type": "object",
                    "properties": {
                        "arg": { "type": "string" }
                    },
                    "required": ["arg"]
                }
            }
        }
    }
```

Then declare it in `config.json`:

```json
"tools": [
  { "name": "your_tool", "type": "domain",
    "module": "domains.your_name.tools.your_tool" }
]
```

The orchestrator imports the module and calls `build(domain)` once
per cycle to get the `ToolEntry`. The fn is called with parsed args
when the LLM invokes it.

## Step 5 — try a dry-run

```bash
dndlab inspect --domain your_name    # validates config + lists movements
dndlab dry-run --domain your_name    # cycle without LLM call
```

The dry-run runs the orchestrator with `agent` disabled — exercises
the rest of the movements (build_field, structural_check, etc.) so
you catch wiring errors without spending API credits.

## Step 6 — first real cycle

```bash
dndlab run --domain your_name
```

Expect a TENSION_REMAINS verdict on the very first run if you
haven't given the lab enough context. Don't overinterpret the first
cycle's quality — the lab improves across cycles, not within one.

## Common pitfalls

- **Tensions phrased as questions instead of claims** — "What is X?"
  doesn't carry the structural commitment that "X is Y, but observation
  shows Z" does. The lab needs the commitment to falsify.
- **Anti-patterns too abstract** — "don't be sloppy" is useless. List
  concrete errors with code/text examples.
- **Forgetting the bicono** — the bicono is the filter, not ornament.
  If your domain doesn't have natural dipoles, work on identifying
  them before writing the context.
- **Over-listing tools** — every tool inflates the prompt. Start with
  filesystem + python + bash, add specialized tools only when the
  agent's logs show it would benefit.

## Reference

Both shipped domains are the canonical reference:

- `domains/physics/` — numerical, theory-driven, tools heavy on
  computation
- `domains/editorial/` — semantic, corpus-driven, tools heavy on
  text analysis

Read both, copy whichever is closer to your domain, adapt.
